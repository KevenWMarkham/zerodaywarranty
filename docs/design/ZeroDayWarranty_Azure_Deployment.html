Zero Day Warranty Agentic Scenario

Azure deployment design — self-contained in resource group Agentic-Automotives
From: Keven Markham · Deloitte's Microsoft Technology & Services Practice Date: June 2026 Substrate: dev / RnD (APEX-M) Companion to: Architecture · Swim Lanes · Persona Portals · Experts Panel
Executive summary · the one paragraph

The Zero Day Warranty solution deploys self-contained into a single new resource group, Agentic-Automotives (eastus2), in the <subscription-name> subscription. Every resource is project-owned and created fresh — its own Container Registry, Azure OpenAI account, Postgres flexible server, Container Apps environment, Key Vault, managed identity, and observability — with no shared dependencies on any other resource group. The agent runs on a project gpt-4.1-mini deployment, the medallion lands in Postgres schemas zdw_bronze / zdw_silver / zdw_gold, and the hash-chained audit ledger is an append-only table signed by a key in the project Key Vault. This document is the deployment design; the matching reference Bicep lives under infra/.

Naming follows the RnD pattern

Resource names mirror the conventions of the shared RnD environment (e.g. acr…, aoai-…, cae-…, pg-…) but are project-owned. Globally-unique names (ACR, AOAI, Postgres) take a deterministic 6-char suffix from uniqueString().

02Resource topology

Everything lives in Agentic-Automotives. The only thing outside the RG is the Microsoft 365 tenant surface (Teams) for the human approval. Nothing leaves the tenant.

05Data plane · medallion on the project Postgres

The project's own Postgres flexible server is the lakehouse substrate; the medallion maps to schemas in the project database zdw. The production target remains Microsoft Fabric OneLake (Architecture §02) — schema names and the per-VIN Gold contract carry over unchanged.

LayerPostgres objectContents
Bronzezdw_bronze.*raw landing of the four domains (build, warranty, quality, telemetry)
Silverzdw_silver.*VIN-conformed, typed, deduplicated canonical tables
Goldzdw_gold.g_vehicle_root_causeper-VIN joinable view across all four domains
Auditzdw_gold.audit_ledgerappend-only 14-field decision rows · prev_link hash chain

The audit table is append-only: UPDATE/DELETE are blocked by a trigger (WORM-style). The HMAC signing key lives in kv-zero-day-warranty/audit-ledger-signing-key. DDL is in infra/scripts/postgres-schemas.sql.

Capacity dependency

Creating a new Azure OpenAI account + deployments requires available model quota for the subscription in eastus2. If quota is constrained, request an increase or pin the deployments to a region with capacity. The Experts Panel (FinOps / Capacity) tracks this as a deploy-time dependency.

08Deploy sequence

The reference Bicep is subscription-scoped: one pass creates the RG and every project resource. Images and schema are operator steps around it.

1
Authenticate & select subscription.az login --tenant <tenant-id> && az account set -s <subscription-id>
2
Deploy the project (creates RG + ACR + AOAI + Postgres + Container Apps env + KV + identity + apps + RBAC).az deployment sub create -l eastus2 -f infra/main.bicep -p infra/main.parameters.json -p pgAdminPassword='<strong-secret>'
3
Build & push images to the project ACR.az acr login -n <acrName from output> ; docker push <acr>.azurecr.io/zdw/orchestrator:0.1.0 (+ mcp-warranty, mcp-ledger)
4
Create the medallion schemas + append-only audit table.psql "$(az keyvault secret show --vault-name kv-zero-day-warranty -n database-url --query value -o tsv)" -f infra/scripts/postgres-schemas.sql
5
Restart the apps to pull images, then smoke test.curl https://ca-zdw-orchestrator.<env>.eastus2.azurecontainerapps.io/run → expect 24 audit rows, chain VERIFIED

10What this design is — and is not

  • Fully self-contained. One resource group owns every resource; no shared dependencies. Tear-down is a single RG delete.
  • RnD / dev substrate. Public endpoints + RBAC. Private endpoints, customer-managed keys, Entra DB auth, and HA/geo-redundancy are noted as hardening — see the Experts Panel for the full gap log.
  • Fabric-portable. The Postgres medallion is the RnD substrate; the production target is Microsoft Fabric OneLake. The schema names and the per-VIN Gold contract carry over unchanged.
  • Not yet applied. This is the deployment design and reference IaC for review — no resources have been created.
