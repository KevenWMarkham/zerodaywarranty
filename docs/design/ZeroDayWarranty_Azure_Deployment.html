Zero Day Warranty Agentic Scenario

Azure deployment design — resource group Agentic-Automotives
From: Keven Markham · Deloitte's Microsoft Technology & Services Practice Date: June 2026 Substrate: dev / RnD (APEX-M) Companion to: Architecture · Swim Lanes · Persona Portals
Executive summary · the one paragraph

The Zero Day Warranty solution deploys into a new resource group, Agentic-Automotives (eastus2), in the Global_RnD_Agentic_MERCH subscription. It creates only its own project-scoped resources — a Key Vault, a managed identity, and three Container Apps (orchestrator + two MCP servers) — and reuses the shared RnD platform already in rg-iot-visionkit: the ACR, the Container Apps environment, the Azure OpenAI account, and the Postgres flexible server. The agent runs on gpt-4.1-mini, the medallion lands in Postgres schemas zdw_bronze / zdw_silver / zdw_gold, and the hash-chained audit ledger is an append-only Postgres table signed by a key in the project Key Vault. This document is the deployment design; the matching reference Bicep lives under infra/.

On-behalf-of (production)

In the RnD substrate the agent runs under its own managed identity. In a client production deployment the downstream tool calls run on-behalf-of the invoking Quality analyst (Entra OBO), so data access is scoped to the originating user — see the Architecture document, Agent Framework card.

05Data plane · medallion on Postgres

This RnD environment uses the shared Postgres flexible server as the lakehouse substrate. The medallion maps to per-project schemas; the production target remains Microsoft Fabric OneLake (Architecture §02).

LayerPostgres objectContents
Bronzezdw_bronze.*raw landing of the four domains (build, warranty, quality, telemetry)
Silverzdw_silver.*VIN-conformed, typed, deduplicated canonical tables
Goldzdw_gold.g_vehicle_root_causeper-VIN joinable view across all four domains
Auditzdw_gold.audit_ledgerappend-only 14-field decision rows · prev_link hash chain

The audit table is created append-only: UPDATE/DELETE are revoked from the app role, mirroring a WORM policy. The HMAC signing key lives in kv-zero-day-warranty/audit-ledger-signing-key. DDL is in infra/scripts/postgres-schemas.sql.

06Model mapping · Azure OpenAI

The agent calls the shared aoai-apex-demo account. The recommended chat deployment is gpt-4.1-mini (matches the Sonepar pattern — balanced cost/quality); embeddings use text-embedding-3-small for the NeMo-Retriever-style RAG step.

Manifest model pins

The agent manifests under service/AXLE-WARRANTY-01/agents/*/agent.yaml carry reference model pins. At deploy time the pin resolves to the AOAI deployment name via the AOAI_CHAT_DEPLOYMENT=gpt-4.1-mini app setting — the APEX pattern of binding the model at deployment, not in source.

08Deploy sequence

The reference Bicep is subscription-scoped: it creates the RG and the project resources in one pass. The data-plane and image steps are operator actions around it.

1
Authenticate & select subscription.az login --tenant 2da40318-46be-402c-ba75-cfb1f656567d && az account set -s 3c8215d1-350c-4b83-bb7f-b5d26b4280f6
2
Build & push images to the shared ACR.az acr login -n acrvisionkit4459 ; docker push acrvisionkit4459.azurecr.io/zdw/orchestrator:0.1.0 (+ mcp-warranty, mcp-ledger)
3
Deploy the project (creates RG, KV, identity, RBAC, container apps).az deployment sub create -l eastus2 -f infra/main.bicep -p infra/main.parameters.json
4
Seed Key Vault secrets (db-url, signing key, AOAI endpoint, Teams webhook).az keyvault secret set --vault-name kv-zero-day-warranty -n database-url --value "$(az keyvault secret show --vault-name kv-home-agent -n database-url --query value -o tsv)"
5
Create the medallion schemas + append-only audit table.psql "$DATABASE_URL" -f infra/scripts/postgres-schemas.sql
6
Smoke test — run the chain, confirm 24 audit rows + verified hash chain in zdw_gold.audit_ledger.curl https://ca-zdw-orchestrator.<env>.eastus2.azurecontainerapps.io/run → expect 24 rows, chain VERIFIED

10What this design is — and is not

  • RnD / dev substrate. Public endpoints + RBAC. Private endpoints, customer-managed keys, and Entra DB auth are noted as hardening for stage/prod.
  • Cost-frugal by reuse. Only a Key Vault, an identity, and consumption-billed Container Apps are new; ACR (Basic), AOAI, the Container Apps env, and Postgres are shared.
  • Fabric-portable. The Postgres medallion is the RnD substrate; the production target is Microsoft Fabric OneLake. The schema names and the per-VIN Gold contract carry over unchanged.
  • Not yet applied. This is the deployment design and reference IaC for review — no resources have been created.
