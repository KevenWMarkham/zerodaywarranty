Roadmap, sprint backlog & orchestrator — with built / deployed / tested gates
From: Keven Markham · Deloitte's Microsoft Technology & Services Practice Date: June 2026 Backlog: backlog/roadmap.yaml Orchestrator: zdw roadmap · sprints · checklist
Executive summary · the one paragraph

Delivery is planned as four phases / twelve sprints, driven by a single backlog (backlog/roadmap.yaml) and a sprint orchestrator (zdw roadmap · sprints · checklist). Phases P1–P2 (the reference build, the design pack, and the design validation) are complete; P3 stands up the self-contained Azure deployment; P4 reaches pilot readiness. Every deployable component is governed by three gates — built → deployed → tested — and is only counted complete when all three are checked. The orchestrator renders these as a live checkbox matrix, so "done" is provable, not asserted.

01How this plan is driven

The backlog is the single source of truth. The orchestrator reads it and renders progress; the gap log from the Experts Panel is folded in as sprint stories. Nothing is "done" by narration — a deployment is done only when its three gates are checked.

done story / sprint complete in progress underway todo not started [x] built [x] deployed [x] tested = a validated deployment

Definition of "validated"

Built = the artifact exists (image pushed, Bicep what-if clean). Deployed = applied to Agentic-Automotives. Tested = the post-deploy check passed (e.g. zdw validate against the live resource, or the smoke test). A component is validated only when all three are true.

Eval-harness design (S4-3 → S10)

A golden-case eval set (eval/cases/*.yaml) pins expected outcomes for synthetic scenarios — suspect lot, hot station, significance, headline financials. An offline runner executes the chain per case and scores: attribution correctness, significance after multiple-testing correction, confidence calibration, and audit-chain integrity. Sampled production audit rows are promoted into new eval cases (the APEX feedback loop), so the eval set grows with real decisions. Gate: eval must pass before any prompt/model change merges.

Audit durability + private networking (S4-4 → S11)

Durability: stream sealed audit rows from zdw_gold.audit_ledger to immutable (WORM) blob storage and echo to Microsoft Purview; verify the hash chain on export. Networking: private endpoints on Key Vault, ACR, Azure OpenAI, and Postgres; internal-only ingress on the MCP apps; the orchestrator reachable only through the front door. Replace the Postgres password connection string with Entra DB authentication via the managed identity.
